## 一、业务概览

### 产品迭代线
| 版本 | 时间节点 | 核心内容 |
|------|----------|----------|
| Jobs 2.0 | 2024 Q1 | 基础口语课框架，模板化开发 |
| Jobs 2.1 | 2024 Q2 | 模板优化 + 专项 + 纠音功能 |
| Jobs 2.2 | 2024 Q3 | 自研模型接入、翻牌子迭代 |
| Jobs 2.4/2.5 | 2024 Q4 | GPT + 自研模型双轨、灵活打断 |
| Food 系列 | 2024 H2 | 对话课、表达课、综合运用课 |
| L3 系统课 | 2025 Q1 | 正式售卖，用户量扩大 10 倍 |
| L4 系统课 | 2025 Q3 | 新模板、新内容持续迭代 |

### 滚动生产流程
```
W2: 产品拉群，明确 owner
W3: 教研剧本/Prompt 定稿
W4-W6: 研发开发 + PE 联调
W7: 教研配置
W8-W9: 测试 + 走查
W10: 上线 + 用户上课
W11+: 滚动迭代（P0 问题修复）
```

## 二、核心技术问题与解决方案

### 1. 低端设备适配
**问题**：iPad 6/iPhone 7p/iPhone 6s 等设备 CPU 打满、掉帧严重

**解决方案**：
- 降低帧率（Rive/Spine 动画）
- 视频资源分档（urlKey 方案）
- 图片位深 32→16，压缩体积
- 全屏大老师不用时及时移除
- 关闭 Android 10 以下硬件加速

### 2. 断线重连与回放
**问题**：用户退出直播、网络断连后无法正常恢复

**解决方案**：
- Server 维护 state 数据，保证重建房间数据继承
- 客户端识别直播断连情况，清洗回放数据
- 交互彩蛋使用点信息（input + trigger）恢复用户拖拽过程
- 回放支持按 chatloop 回溯点

### 3. 动画性能优化
**问题**：Spine/Rive 资源加载慢、内存占用高

**解决方案**：
- 删除 Spine 中无用资源数据
- Rive 升级到 6.0，确认适配性后使用
- 异步加载 Rive，指令排队处理
- 老师骨骼数量上限规范（动画产出规范文档）
- 前端做预加载优化

### 4. AI 对话稳定性
**问题**：PE 输出不稳定、GPT 不说话、流程卡住

**解决方案**：
- PE 必须在评审阶段保证用例正常
- Server 增加卡住检测告警（30s 卡住直接告警前端/客户端）
- 灵活打断需要配合回声消除方案
- TPM 超限时对应新错误码，配置不重试

### 5. 音视频相关
**问题**：TTS 口型、语音识别、视频课件等

**解决方案**：
- 音素配置校验（时长对齐）
- 客户端本地播放器替换（YLPlayer）
- 透明视频预乘处理解决锯齿问题
- 中台实时录制增加重试机制

## 三、跨团队协作问题

### 1. PE（Prompt Engineering）协作
| 问题 | 频率 | 应对措施 |
|------|------|----------|
| PE 流程不稳定导致联调阻塞 | 高频 | PE 在 W3 定稿，W5 初版，W9 稳定 |
| PE 临时改动影响提测 | 高频 | 提测前冻结 PE 改动，只改话术不改流程 |
| PE 与 Server 数据结构不对齐 | 中频 | 评审阶段确定，避免后期翻工 |

### 2. 动画资源交付
| 问题 | 频率 | 应对措施 |
|------|------|----------|
| 正式资源给得晚 | 高频 | 上线 deadline 前 2 天交付，Rive 更早 |
| 资源替换后没通知研发 | 中频 | 动画改动必须周知负责的前端 |
| 占位资源与正式资源差异大 | 中频 | 占位资源需提前对动画段 |

### 3. B 端（教研后台）问题
| 问题 | 频率 | 应对措施 |
|------|------|----------|
| 配置相互覆盖 | 高频 | 后台迭代支持操作日志 |
| 测试/线上配置不同步 | 中频 | 测试环境改动同步线上回测验证 |
| 无用配置 key 过多 | 中频 | 定期梳理清理 |

### 4. 产品走查
| 问题 | 频率 | 应对措施 |
|------|------|----------|
| 走查问题描述模糊 | 高频 | 客户端测试包增加 loop 展示，要求截图 |
| UI/PRD 不一致 | 中频 | 按 PRD 开发逻辑，UI 稿只对样式 |
| 走查发现临时需求 | 中频 | 流程截止周四，周五过风险 |

## 四、值班与线上问题

### 值班机制演进
```
2024 初: 轮转值班
2025 中: 滚动生产专人跟进，不再轮转
        - 滚动生产 → 前端负责人分配
        - 滚动迭代 → 服务端负责人分配
        - 其他 zerd → 客户端负责人分配
```

### 线上问题处理流程
```
1. 第一优先级：让用户尽快上课完课（不要陷入排查）
2. 需要多方合作 → 负责人拉群，zerd 承担故障指挥角色
3. 单人可解决 → zerd 账号回复结论或转交负责人
4. 故障级别判断：
   - 购买/mission入口/上课/报告 阻断 → 发起故障流程
   - 体验类问题 → 评估影响后排期修复
```

### 高频线上问题
| 问题类型 | 占比 | 处理方式 |
|----------|------|----------|
| 收音识别失败 | 最多 | 值班提效方案讨论中 |
| 直播卡住/30s 无响应 | 中等 | 告警直接发给客户端/前端 |
| 交互彩蛋异常 | 中等 | 复盘后沉淀排查方案 |
| 配置问题 | 中等 | B 端增加校验 |

## 五、关键经验总结

### 做得好的
1. **模板化开发**：环节抽象 + 剧本配置，降低重复开发成本
2. **提效工具**：本地联调工具、mock widget、断点调试支持
3. **监控告警**：Octopus 实时看板、卡住告警分级
4. **流程规范**：滚动生产/迭代流程逐步稳定

### 待改进的
1. **PE 稳定性**：仍是最大的不确定性来源
2. **资源交付节奏**：动画/音效排期经常被忽略
3. **低端设备**：16% 用户占比，长期需要兼容
4. **测试环境稳定性**：重构期间问题频发

### 核心协作原则
> 1. **合作问题在群里沟通，不要私聊**
> 2. **测试环境改动同步线上回测验证**
> 3. **动画资源上线 deadline 前 2 天交付**
> 4. **PE 提测前冻结改动，只改话术不改流程**
> 5. **走查问题周四解决完，周五过风险**

## 六、对接手团队的启示

### 需要重点了解的
1. **滚动生产/迭代当前状态**：哪些 Unit 在进行中？
2. **PE 团队当前人力**：是否有瓶颈？配合模式？
3. **低端设备策略**：当前占比？是否有限制计划？
4. **AI 模型选型**：自研 vs GPT 的使用场景？
5. **值班现状**：问题响应及时性？痛点？

### 可以复用的
1. 滚动生产流程已成熟，可直接复用
2. 提效工具（mock、调试）已有基础
3. 监控告警体系已建立
4. 走查/问题跟进流程有文档规范

### 需要警惕的
1. PE 改动是最大的不稳定因素
2. 动画交付时间经常 delay
3. B 端配置问题容易遗漏
4. 低端设备兼容是长期战役
