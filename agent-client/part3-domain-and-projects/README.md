# Part 3：公司内部领域与项目总结

> **TL;DR**：这一部分的目标不是再讲一遍技术栈，而是把前两部分的通用能力落到真实业务里。它最终要沉淀的不是“项目做了什么”，而是 **领域语言、系统边界、关键流程、典型取舍、排障经验** 这些能帮助新人快速进入业务的内容。

## 这一部分要解决什么问题

很多团队的技术学习资料都有一个共同问题：

- 技术基础写得不少
- 业务资料也零散存在
- 但新人真正开始接项目时，还是不知道系统为什么这样设计

原因通常不是“资料太少”，而是缺少一套把技术能力和领域知识连接起来的文档结构。

Part 3 的作用，就是补上这层连接：

- 把业务地图说清楚
- 把系统边界说清楚
- 把项目里的关键取舍说清楚
- 把老员工脑子里的排障经验和例外规则沉淀下来

## 为什么这一部分必须依赖真实项目

领域文档最怕两种写法：

1. 只有抽象概念，没有真实流程
2. 只有项目流水账，没有领域抽象

所以这部分不能空想，必须依赖实际项目材料来写。  
也正因为这样，当前阶段更合理的做法是先把框架搭好，等你给具体项目后再填实。

## 每篇项目总结应该怎么写

项目总结最容易失去价值的写法，是只写“我们做了什么”。  
真正有复用价值的项目总结，至少要回答下面这些问题：

### 1. 当时要解决什么业务问题

不要直接从技术方案写起。  
先把业务目标和业务约束说清楚，否则后面的技术取舍就没有判断上下文。

### 2. 为什么选了这个方案

一个方案的价值，不只在于“最后做成了”，更在于：

- 当时有哪些备选路径
- 为什么没有选另外的方案
- 这些选择背后的前提是什么

### 3. 系统边界是怎么划的

很多复杂项目并不是实现难，而是边界难：

- 哪个系统负责什么
- 哪个状态在哪个系统维护
- 哪个动作必须同步，哪个可以异步
- 出错后由谁兜底

### 4. 最容易踩的坑是什么

这部分往往比“最终架构图”更值钱。  
因为真正帮助新人上手的，通常不是理想路径，而是例外路径和失败路径。

### 5. 如果下次再做，会如何改

这是把项目经验转成团队能力的关键。  
没有这一步，项目总结就很容易退化成“历史记录”，而不是“未来可复用的方法”。

## 单篇项目文档模板

建议每个领域项目都尽量遵循同一模板：

1. TL;DR
2. 业务目标与背景
3. 业务流程概览
4. 领域对象、状态与规则
5. 系统边界与上下游关系
6. 技术方案与取舍
7. 常见异常流程与排障路径
8. 本项目和 Part 1 / Part 2 的关联知识点
9. 如果重做一次，会怎样改
