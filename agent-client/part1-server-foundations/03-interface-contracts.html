接口契约

HTTP、RPC 与服务端协作边界

接口为什么是服务端能力的分水岭

刚开始做服务端时，很容易把接口理解成"把数据吐出去"。

但接口一旦被别人依赖，它就不再只是你本地代码的一部分，
而会变成一个协作边界。

接口被依赖后，难点在哪

  • 别人能不能正确理解它
  • 它失败时是否容易判断原因
  • 它被重试时会不会把状态打坏
  • 它后续演进时会不会破坏旧调用方

接口最难的部分，通常不是第一次发布，而是第二次修改

接口能力的本质

不是"会定义 JSON"
或"会写 thrift 文件"

而是会设计
长期可协作的边界

接口约束的三层框架

协议约束

请求/调用在协议层怎么表达

HTTP: Method、状态码、Header
RPC: 服务名、方法名、异常定义

调用约束

调用方失败时会看到什么

超时、重试、错误响应
幂等要求、异常传播

业务约束

业务上到底在承诺什么

业务错误码、分页结构
字段语义、返回值约定

这三层一旦混在一起，接口就会很快变得难维护

典型混乱场景

HTTP 混乱

HTTP 层永远返回 200
真正错误全塞进业务码
但业务码又同时承载"系统异常"和"业务不允许"
调用方只好靠文案或日志去猜

RPC 混乱

方法签名看起来很简单
但超时、重试、异常传播完全没有明确约定
调用方把它当成本地方法来用
出问题时才发现"网络调用根本不是本地调用"

HTTP

HTTP 在约束什么

HTTP 本质上是一套协议层语义，帮助调用方理解：

  • 你在操作什么资源
  • 你在对这个资源做什么动作
  • 这个动作成功还是失败
  • 失败属于哪一层
HTTP

Method 在表达动作语义

Method语义特点
GET读取天然幂等、可缓存
POST创建或提交动作通常非幂等
PUT整体更新幂等
DELETE删除幂等

如果所有动作都塞进 POST，调用方会失去很多判断能力

HTTP

状态码在表达协议层结果

状态码含义调用方行为
200请求被正常处理解析业务响应
400请求格式或参数不合法检查请求参数
401身份未通过认证重新登录
403身份通过了，但没有权限提示无权限
404资源不存在检查资源 ID
500服务端内部异常可考虑重试或报警
HTTP

典型坏味道

所有请求都返回 200
错误全靠 code != 0 表示
  • 把协议层失败和业务层失败全抹平了
  • 调用方需要自己额外约定一套解释逻辑
  • 观测系统很难快速看出协议层故障
  • 网关层和客户端层的默认行为也失去依据
RPC

RPC 的最大误导性

它太像本地方法调用了

方法签名一摆出来，很容易让人下意识觉得：
"这不就是调个函数吗？传个对象进去，拿个对象出来"

但 RPC 从来不是本地调用

RPC

RPC 天然带着的额外问题

本地调用
调用即到达
失败即异常
无需考虑网络
vs
RPC 调用
网络超时
请求发了，响应没收到
重试行为
框架可能自动重试
异常传播
远端异常如何到调用方
序列化兼容
字段增删的影响
版本不一致
调用方和服务端可能跑着不同版本

RPC 不是把网络调用伪装成没有代价的本地调用，而是用更强类型的方式表达服务间协作

RPC

Thrift 定义示例

exception TConanMissionOralBizException {
    1: i32 code;
    2: string msg;
}

struct TCreateLessonUserMissionRequest {
    1: required i64 oralLessonMissionId;
    2: required i64 userId;
    3: required i32 displayDate;
    4: optional i32 expireDate;
    5: optional string ext;
}

哪些字段必需、哪些可选、业务异常如何单独建模

共通问题 1

成功与失败如何分层

层次HTTP 表现RPC 表现典型含义
协议层 200/400/401/403/404/500 方法是否调用成功 请求有没有被正确接收和处理
调用层 超时、连接失败、网关失败 TException、超时、下游不可用 这次远程调用本身是否可靠完成
业务层 bizCode、业务 message 业务异常、业务返回结构 业务是否允许、状态是否合法
共通问题 1

关键认知

HTTP 成功，不代表业务成功
RPC 抛异常，也不一定都代表系统故障
  • 下单接口返回 200，但业务码提示"库存不足"
  • RPC 方法返回了结构体，但业务字段提示"当前状态不允许操作"
  • RPC 方法抛出框架级异常，可能是超时或网络故障

如果这三层不分开，调用方就很难做正确的决策

共通问题 2

超时、重试与幂等

一个常见的服务端现实：

调用方发起请求 -> 服务端已开始执行 -> 调用方超时
调用方自动重试 -> 同一动作再次执行

如果接口没有幂等保护，就会出现重复执行问题

共通问题 2

幂等保护的是什么

系统状态不能因为
一次超时和一次重试
而被重复修改

设计接口时，必须提前想清楚：

  • 调用方会不会重试
  • 超时后服务端有没有可能已经成功
  • 相同请求再次到来时，是重复创建、重复扣款，还是安全返回旧结果
共通问题 2

幂等处理示例

public TCreateLessonUserMissionResponse createLessonUserMission(...) {
    TCreateLessonUserMissionResponse response = new ...;
    try {
        lessonUserMissionService.createOrUpdateStatusToValidIfExist(...);
    } catch (DuplicateKeyException e) {
        response.setSuccess(false);
        response.setExisted(true); // 告诉调用方：已存在
        return response;
    }
    response.setSuccess(true);
    return response;
}

重复请求不是再创建一次，而是返回一个可解释的结果

共通问题 2

高风险的幂等场景

下单 支付回调 创建资源 消息消费后状态更新

边界：

  • 不是所有接口都天然幂等
  • 查询类接口通常更接近幂等
  • 写接口往往需要显式设计幂等键、业务唯一键或状态判断

幂等保护的不是"代码优雅"，而是"系统状态不被调用方行为污染"

共通问题 3

兼容性与演进

接口一旦发布，最危险的时刻通常不是第一次上线，
而是第一次修改
  • 旧调用方还在不在
  • 新旧版本会不会并存
  • 字段变化会不会造成静默错误
  • 默认值变化会不会改变旧逻辑
共通问题 3

最常见的兼容风险

删除字段 改字段语义 改默认值
改枚举含义 改异常表达方式
改语义，往往比改名字更危险

因为改名字很容易在编译期被发现
但语义变化往往会"悄悄错"，直到线上行为异常才暴露
共通问题 3

最小兼容性 Checklist

  • 新增字段通常比删除字段安全
  • 改语义比改名字更危险
  • 默认值变化要非常谨慎
  • 枚举扩展要考虑旧调用方如何处理未知值
  • 异常或错误码变化要考虑老逻辑是否还认识它

兼容性不是"发布时顺手看一下"的事情
而是接口设计第一天就该带着的约束

共通问题 3 · 真实案例

最痛的教训：一个交易系统，两套数据结构

AI课订单
父子订单结构
Order (父订单)
  └─ SubOrder (子订单)
      └─ SubOrder
      └─ ...
vs
同一个交易系统
Wares订单
订单-订单项结构
Order (订单)
  └─ OrderItem (订单项)
      └─ OrderItem
      └─ ...
为什么会这样？
第一次定义时没有统一设计 → 各业务方开始依赖 → 字段语义、查询方式、下游适配全不一样 → 再想统一，改不动了

接口结构定义的第一次，往往就是最后一次机会

实践

一个坏接口是怎么长出来的

协议层和业务层混在一起

所有 HTTP 错误都返回 200
所有 RPC 失败都只抛一个笼统异常
业务错误和系统错误没有分层

短期需求推着接口漂移

所有动作都先用 POST
RPC 方法签名跟着业务短期诉求频繁变化
字段名和语义不断加历史包袱

返回结构风格不统一

分页结构每个接口都不一样
同一概念在不同接口里叫法不同
同样的失败，表达方式不同

只通知当前调用方

接口有多个调用方时
"我已经通知到前端了"
并不代表真的安全

实践

好接口的最小检查清单

1这个接口的资源或方法语义清楚吗？
2HTTP 方法或 RPC 方法名是否准确表达动作？
3协议层/调用层/业务层成功失败是否分层？
4调用方超时后会不会重试？重试后状态会不会被重复修改？
5这个接口是否考虑了幂等？
6字段、默认值、枚举、异常模型后续如何兼容演进？
7返回结构是否稳定、一致、可预期？

学完后应该获得的判断力

看接口时

不再只盯字段

会区分 HTTP/RPC 约束
协议层/业务层失败
调用行为/数据结构问题

设计接口时

会先想调用方接下来会做什么

对方会不会重试
失败后该看到什么信息

判断接口质量时

不再只看"能不能联调"

是否长期可维护
后续演进会不会搞崩旧调用方

核心 Takeaway

接口设计
本质上是在设计
协作成本

HTTP 和 RPC 的共同角色

  • 给调用方稳定预期
  • 给系统划清边界
  • 给后续演进留出余地

真正要做到的是：
把协议、调用、业务这三层约束拆清楚
并提前替调用方想清楚失败、重试和演进时会发生什么
