# 交付闭环：测试、可观测性与安全上线

> **TL;DR**：会写功能，只是服务端能力的一半。另一半是知道怎么验证它是对的、怎么在出问题时快速发现、怎么在运行时保护它不被拖垮、以及怎么安全地把它交付到生产环境。这一篇就是要把"会开发"变成"能交付"。

---

## 功能写完了，然后呢

前六篇走到这里，系统已经不再简单了：

- 主状态落在数据库里
- Redis 承接了读路径数据和部分运行时结构
- consumer 在实时追赶数据库的变化
- job 在周期性校验和修正

第六篇收尾的时候说过一句话：系统的复杂度不是一次设计出来的，而是在解决真实问题的过程中一层一层长出来的。

但复杂度长出来之后，紧接着就是一个更尖锐的问题：

**你怎么知道它现在是对的？出了问题你怎么发现？发现了你怎么定位？修好了你怎么安全上线？**

这四个问题，就是第七篇要回答的。

在展开之前，先建立一个认知：**大多数线上事故，不是因为代码逻辑写错了，而是因为没人知道它已经错了。** 一个 Redis 同步漏了、一个下游接口慢了、一个配置改错了——这些事情发生的时候，如果系统没有任何方式告诉你"我出问题了"，你就只能等用户投诉、等老板问、等业务方来找你。

所以第七篇的主线不是"这些工具该怎么用"，而是：**怎么建立一套最小闭环，让你在功能之外，同时具备验证、保护、观测和安全交付的能力。**

---

## 第一部分：测试——不是为了通过，而是为了建立信心

### 1. 功能做完后，至少该验证什么

很多人对测试的理解还停留在"领导要求写单测"或者"覆盖率要到多少"。但在真实服务端开发里，测试要回答的核心问题其实只有一个：

**这次改动上线之后，我敢不敢不盯着？**

如果答案是"不敢"，说明你对自己的改动没有足够的信心。而测试就是建立这种信心的最直接手段。

### 2. 三个层次，从近到远

**第一层：核心逻辑的单测。**

这不是为了覆盖率，而是为了一个非常实际的目的：**下次别人改这段代码的时候，能立刻知道有没有改坏。**

比如商品服务里，商品属性的校验逻辑、价格的计算规则、状态流转的合法性判断——这些地方如果没有单测，每次改完都只能靠手动联调来验证，效率极低，而且很容易漏。

**第二层：接口级的联调与集成验证。**

单测保证的是一个方法内部的逻辑正确，但服务端的问题往往出在"多个模块连起来之后"。所以你至少要验证：

- 接口入参合法和非法时，返回是否符合预期
- 上下游依赖正常和异常时，你的服务是否能正确处理
- 数据库和 Redis 的状态变化是否一致

这一层不需要搞一套完整的 e2e 测试体系，但至少要在测试环境里把主链路走通一遍。

**第三层：边界场景和失败路径。**

这是最容易被忽略、也最容易出事的一层。

- 传了空值会怎样？
- 同一个请求重复发了两次会怎样？
- 数据库写成功了但消息发送失败了会怎样？
- 并发写同一条商品数据会怎样？

大多数线上问题不是因为"正常流程没走通"，而是因为"异常流程没人想过"。

### 3. 用商品案例串一下

假设你改了商品属性的写入逻辑，上线前至少该验证这几件事：

- 写入一条新属性，数据库里的记录对不对
- 写入之后，consumer 有没有把 Redis 同步上去
- 重复写入同一个属性，会不会产生脏数据
- 写入一个不合法的属性值，接口是否返回了明确的错误
- 写入成功后，查询接口能不能正确返回最新数据

这五件事不需要多复杂的测试框架，但如果一件都没验证就上线，出问题只是时间问题。

## 第二部分：稳定性——不是事后补的，而是写代码时就该想

### 1. 逻辑没错，为什么上线还是会出问题

这是很多刚接触服务端的同学最困惑的地方：**我的代码逻辑明明是对的，为什么线上还是出了问题？**

答案通常不是"代码写错了"，而是"运行时条件超出了你的预期"。

- 你调的下游接口平时 50ms 返回，今天突然 5 秒才返回
- 你的重试逻辑在正常情况下没问题，但下游挂了之后，重试把对方彻底打死了
- 你的服务本来只处理正常流量，但某个时刻流量突然翻了 10 倍
- 一个慢查询占住了数据库连接池里的连接，导致其他正常请求全部排队超时

这些都不是 bug，但它们造成的影响可能比 bug 严重得多。

### 2. 四个必须提前想到的切面

**超时：你调别人，对方慢了怎么办？**

任何一次外部调用——RPC、HTTP、数据库查询、Redis 读写——都必须设超时。不设超时意味着：一旦对方慢了，你的线程就会被无限期占住，连带着你自己的服务也开始变慢。

在商品服务里，查一次商品详情可能要调价格服务、库存服务、类目服务。如果其中任何一个没设超时，一个下游的慢就能拖垮你整个接口。

**重试：重试一次没问题，重试三次可能把下游打挂。**

重试的本意是"偶尔失败了再试一次"，但如果所有调用方都在同一时间重试，下游收到的压力就不是"原来的两倍"，而可能是"原来的十倍"。

所以重试不能无脑加。至少要考虑：

- 这个操作是幂等的吗？重复执行会不会产生副作用？
- 重试的间隔是不是递增的？（避免同一时刻大量请求涌入）
- 重试次数有没有上限？

**降级：核心链路挂了，是整个接口 500 还是局部降级？**

在商品详情页里，价格是核心信息，但推荐商品列表不是。如果推荐服务挂了，更合理的做法是返回商品详情但不返回推荐列表，而不是整个接口报错。

降级的核心思想是：**区分"必须成功"和"可以容忍失败"的部分，失败时只丢掉非核心的。**

**资源保护：一个慢请求拖垮所有请求。**

服务端的线程池、数据库连接池、Redis 连接池都是有限资源。如果一个慢查询长时间占住连接，其他正常请求就会排队等待，最终所有请求都超时。

这就是为什么需要对资源做隔离和限制：核心接口和非核心接口不要共享同一个线程池；慢查询要有超时熔断；流量突增时要有限流保护。

### 3. 一句话总结

**稳定性问题通常不是你的 bug，而是你没有预想到的运行时条件。** 超时、重试、降级、资源保护，不是"高级功能"，而是服务端代码的基本防线。

## 第三部分：可观测性——出了问题，你该看什么

### 1. 从一个真实场景开始

假设线上有人反馈：某个商品查不到了。

你打开电脑，第一反应是什么？

很多人的第一反应是"翻日志"。但如果你面对的是一个每秒处理几千次请求的服务，翻日志可能意味着在几十万条记录里大海捞针。

更高效的做法是先缩小范围，再精确定位。而这就需要三种不同粒度的观测手段。

### 2. 日志、指标、Trace 分别在回答什么问题

```mermaid
graph LR
    problem[线上问题]
    metrics[指标]
    trace[Trace]
    log[日志]

    problem --> metrics
    metrics --> trace
    trace --> log
```

**指标（Metrics）：告诉你"最近一段时间，系统整体怎么样"。**

- 接口的平均响应时间是不是突然变长了？
- 错误率是不是从 0.1% 跳到了 5%？
- 数据库连接池的使用率是不是接近满了？

指标不回答"某一次请求发生了什么"，但它能快速告诉你"系统是不是出了问题"以及"问题大概在哪个方向"。

**Trace（链路追踪）：告诉你"这个请求经过了哪些节点，在哪里慢了"。**

一次商品查询可能经过了网关、商品服务、价格服务、Redis、数据库。如果整体变慢了，Trace 能帮你看到是哪一跳变慢了，是网关慢、是数据库慢、还是某个下游服务慢。

**日志（Logs）：告诉你"某一次请求具体发生了什么"。**

当你已经通过指标发现了问题方向，通过 Trace 定位到了慢在哪个节点，最后才需要翻日志去看：这次请求的入参是什么、走了哪个分支、异常信息是什么。

### 3. 为什么要按这个顺序

三者不是可以互相替代的，而是在回答不同粒度的问题：

| 工具 | 回答的问题 | 粒度 | 适合什么时候看 |
|---|---|---|---|
| 指标 | 系统整体是否正常 | 粗 | 第一时间判断有没有问题 |
| Trace | 某次请求在链路上哪里出了问题 | 中 | 缩小范围，定位到具体节点 |
| 日志 | 某次请求具体发生了什么 | 细 | 最后确认根因 |

**很多人出了问题只会翻日志，是因为不知道指标和 Trace 能更快帮你缩小范围。** 学会"先看指标判断方向 -> 再看 Trace 定位节点 -> 最后看日志确认细节"，排查效率会有质的提升。

## 第四部分：上线——不只是合并代码

### 1. 从"代码改好了"到"用户用上"，中间还有什么

很多同学对上线的理解是：代码合了、CI 过了、部署了，就算完成了。

但真实的服务端上线，从代码合并到用户真正用上新版本，中间至少还有这些事要做：

```mermaid
graph LR
    code[代码合并]
    check[上线前检查]
    canary[灰度发布]
    monitor[上线后观测]
    rollback[出问题回滚]

    code --> check
    check --> canary
    canary --> monitor
    monitor --> rollback
```

### 2. 上线前该检查什么

代码本身的正确性，测试阶段应该已经验证过了。上线前要检查的是另一类问题：

- **配置**：新加的配置项是否已经在生产环境配好了？
- **依赖**：这次改动依赖的上下游服务是否已经就绪？
- **数据兼容**：如果改了数据结构，老数据能不能被新代码正确处理？
- **发布顺序**：如果有多个服务要一起改，谁先发谁后发？

在商品服务里，一个很典型的场景是：你给商品新增了一个属性字段。如果消费端的 consumer 先上线、但生产端的写入逻辑还没上线，consumer 就会因为拿不到新字段而出现意料之外的行为。

### 3. 灰度发布：不要一步到位

灰度的核心思想很简单：**不要一次性把所有用户切到新版本，而是先让一小部分流量走新版本，确认没问题之后再逐步放量。**

这样做的好处是：如果新版本有问题，影响范围被控制在了一小部分用户，而不是全量用户。

具体怎么分批、怎么切流量，不同团队和平台有不同的实现方式。但核心原则是一样的：小步快走，每一步都确认安全之后再继续。

### 4. 上线后该盯什么

发布完成不等于上线结束。上线之后至少要盯一段时间，重点关注：

- **错误率**：是否出现了新的异常？
- **响应时间**：核心接口的 P99 响应时间是否明显上升？
- **业务指标**：商品创建量、查询量等核心业务指标是否正常？
- **资源消耗**：CPU、内存、数据库连接池是否出现异常波动？

如果这些指标在上线后出现明显异常，说明这次发布可能引入了问题。

### 5. 回滚不是丢人的事

很多新手工程师对回滚有一种心理障碍，觉得回滚意味着"我搞砸了"。

但在服务端工程里，回滚是最基本的安全网，不是失败的标志。**一个敢回滚的团队，比一个"出了问题先硬抗"的团队要安全得多。**

回滚的前提是：你在上线之前就要确认这次发布是可以回滚的。如果涉及数据库结构变更、数据迁移、不兼容的接口改动，就必须提前设计好回滚方案。

## 常见误区

### 1. 测试只关注正常流程

正常流程通常不太会出事。真正出事的都是边界条件：空值、重复请求、并发写入、下游超时。如果测试只覆盖了正常路径，等于只验证了最不可能出问题的部分。

### 2. 把可观测性等同于"打日志"

日志只是可观测性的一个维度。没有指标你不知道系统整体状态，没有 Trace 你不知道问题出在链路的哪个环节。三者缺一不可。

### 3. 稳定性是"后面再加的事"

超时、重试、降级这些东西如果不在写代码的时候就考虑，上线后再补的成本会高得多。而且很多稳定性问题的根因，恰恰是因为"第一版没想到这个场景"。

### 4. 上线等于部署成功

部署成功只是上线的第一步。真正的上线完成是：灰度放量、核心指标正常、观测了足够长的时间窗口、确认没有回退的必要。

## 写在最后

前六篇建立的是"怎么把系统建起来"的能力——从心智模型、请求链路、接口契约、代码组织、数据建模，到异步系统和多层数据。

第七篇补的是另一半：**怎么让它在真实环境里活下来。**

如果说前六篇教你的是造一座桥，第七篇要说的是：桥造好了，你还得知道怎么检测它能不能承重、怎么在桥上装传感器知道它什么时候开始出问题、怎么在出问题的时候让车辆安全撤离。

这两件事合在一起，才是一个服务端工程师"能交付"的最小闭环。

到这里，Part 1 的七篇必修就全部结束了。  
从认知切换到请求链路，从接口契约到代码组织，从数据建模到异步系统，再到稳定性和交付——这条线走完，你应该已经具备了参与服务端开发的基本能力。

接下来进入 Part 2，要解决的就不再是"通用服务端怎么做"，而是"在公司环境里怎么真正把东西做出来、发出去、查清楚"。
