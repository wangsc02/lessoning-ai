Redis、MQ 与异步系统的最小必备认知

数据库承接主状态之后，系统复杂度从哪里继续长出来

数据库接住主状态之后，问题为什么还没结束

第五篇已经讲清楚：主状态落在数据库，事务边界贴着业务动作

但真实服务端很快会继续碰到这些问题：

  • 同一个热点数据被反复读，数据库扛不住
  • 一个主动作之后，还有很多后续责任，同步做下去会让链路越来越长
  • 有些问题不是"事件来了就处理"，而是"过一段时间还得再看一眼"

为什么会出现 Redis、consumer 和 job

主状态以数据库为准，但系统不可能所有问题都靠数据库硬扛

第一部分

先讲 Redis

Redis 不只是缓存
而是在承接读路径和中间结构

Redis

Redis 最常见到底在解决什么问题

在中台商品服务里，Redis 承接的角色：

  • 缓存热点商品数据
  • 承接 skuId → spuId 这类映射
  • 承接价格、类目属性这类读路径数据
  • 承接库存待处理集合这类运行时结构
Redis 通常不是为了替代数据库，而是为了保护数据库和同步链路
Redis

数据库的存储模型 vs Redis 的存储模型

数据库

  • 为主状态落库服务
  • 关心一条记录怎么存、怎么改
  • 关心事务边界怎么收住
  • 保存的是主状态

Redis

  • 为读取路径和运行时处理服务
  • 关心查询要不要少拼装几次
  • 关心映射要不要直接命中
  • 保存的是读模型和中间结构

service / consumer 负责把前者转换成后者

Redis

真正麻烦的不是缓存更新，而是多层数据一致性

DB 主状态
↓
Redis 共享数据层
↓
JVM 本地缓存

一旦变成三层，真正麻烦的是：

  • 主库更新了，Redis 里的数据有没有跟上
  • Redis 跟上了，本地缓存有没有失效
  • 某次消息没消费成功，哪一层开始出现偏差
  • Redis 里多出来的脏 key 和脏 field，谁来收
第二部分

consumer 在这里承担什么责任

不是在做业务
而是在追赶数据库里的最新状态

consumer

为什么写库成功后，还要继续发消息

主库写成功，只代表主状态已经落对，不代表 Redis 和本地缓存已经同步完成

写库链路

  • 事务内完成主状态变更
  • 保证主状态正确提交

异步链路（consumer）

  • 事务提交后接管
  • 把变化扩散到 Redis 和本地缓存
  • 同步失败问题明确落在异步层

消息的作用是把"写对主状态"和"同步其他层"拆开处理

consumer

consumer 的职责：实时追赶

同一个主状态变更之后，系统里至少要做三件事：

主库写对 → Redis 数据跟上 → 本地缓存失效
consumer 的职责不是"把业务办完"，而是"把主状态尽快扩散到其他层"
consumer

即使有 consumer，系统仍然可能出现层间不一致

即使有 afterCommit 发消息、consumer 刷 Redis、broadcast 失效本地缓存：

  • 消息可能漏投、延迟或者重复
  • 消费可能失败或长时间重试
  • 某一层写成功，另一层没有及时跟上
  • Redis 里可能残留历史脏数据

consumer 负责实时追赶，而不是单独保证长期一致性

第三部分

job 为什么不是附属任务，而是主动补完整性

除了实时同步
还要有周期性自查

job

有了 consumer，为什么系统还需要 job

consumer 负责尽量追赶，但不负责证明系统长期绝对正确

一旦系统里有多层数据，就总会有这些残留问题：

  • 某条消息漏了
  • 某次消费失败了
  • 某个 Redis key 多出来了
  • 某个 field 旧了，但实时链路再也不会碰到它

这些问题不会自己消失，系统需要一条完全不依赖"刚刚发生了什么事件"的主动修复链

job

job 在这里做的：对账、回填、修复、删脏数据

主要动作

  • 以数据库里的主状态为准
  • 主动把 Redis 里的数据拿出来对比
  • 一旦不一致，直接回填修正

还要处理

  • 清理 Redis 里多出来的整块脏数据
  • 删除单个多余 field
  • 这已经不是"缓存没命中回源"能解决的

job 最值得强调的不是"定时跑"，而是系统主动把已经出现偏差的数据拉回正确状态

job

job 和 MQ 的关系：不是替代，而是互补

很多团队讨论异步时，会下意识变成：到底该 MQ 还是 job
但真实工程里，很多时候不是二选一，而是搭配

consumer

  • 平时做实时同步
  • 更像实时同步 Redis 和本地缓存

job

  • 漏了、挂了、晚了，做对账、修复和清理
  • 更像周期性主动修复

把它们放在一起看，异步系统才会完整

放在一起看

数据库、Redis、consumer、job 比较

层次/机制更适合什么主要优点主要代价
数据库承接主状态状态集中、事务边界清楚扛不住所有高频读
Redis承接读路径和运行时结构读快、共享数据灵活一致性维护成本上升
consumer实时同步 Redis 和本地缓存让多层数据尽量跟上最新状态可能漏、可能失败
job对账、回填、修复、删脏数据把长期积累的偏差拉回正确状态时效性差、治理成本高
放在一起看

看到系统问题时，先问

  • 这是主状态问题，还是 Redis / 本地缓存这层数据问题
  • 这件事需要实时追上，还是允许周期性修正
  • 这里增加的复杂度，到底换来了什么收益

一旦进入异步系统，常见问题就会变成：

Redis 同步漏掉 Redis 多出脏数据 本地缓存失效不及时 回填链路复杂
常见误区

五个典型问题

把 Redis 当成万能数据库

通常只会把主状态和短期状态混在一起

把 MQ 当成"发出去就算完成"

发消息只是把后续动作交给了另一个推进机制

把 job 当成"最后兜底随便扫一下"

job 如果没有明确边界和幂等设计，往往会成为新的风险源

把异步理解成天然更快、更简单

异步只是让当前链路轻一点，但会把复杂度转移到运行时

只会从组件名选型，不会从约束选型 — 真正应该先判断的不是"上 Redis 还是 MQ"，而是这件事要实时做、事件做，还是定时补

写在最后

系统的复杂度不是一次设计出来的
而是在解决真实问题的过程中一层一层长出来的
  • 数据库扛不住高频读 → 引入 Redis
  • Redis 和数据库之间需要同步 → 引入 consumer
  • consumer 不能保证长期正确 → 引入 job

每一步都有合理的理由，但每一步也都在给系统增加新的代价

核心 Takeaway

不存在一个"加了就变好"的组件

只存在"当下这个约束
用哪种方式换来最小代价"的权衡
