第一层

CRUD 万能论

Controller → 接收请求，返回响应
Service     → 写业务逻辑
Storage    → 操作数据库
"服务端真简单，不就是把前端传过来的数据存到数据库，再查出来返回吗？"
第一层

什么时候撞墙？

  • 一个 Service 方法越写越长，几百行塞在一起
  • 改一个功能要动好几个地方，牵一发而动全身
  • 不同接口在重复写类似逻辑，但又不完全一样
  • 出了问题很难定位，代码全糊在一起

三层架构只是代码的物理分层
并不能帮你管理业务复杂度

第二层

架构模式的困惑期

PO

Persistent Object · 数据库表的映射对象

DO

Domain Object · 领域对象，承载业务逻辑

DTO

Data Transfer Object · 跨层数据传递

BO

Business Object · Service 层业务对象

"一个数据从数据库查出来，要转 PO → DO → DTO → VO……这不是自找麻烦吗？"
第二层

这些"繁琐"解决了什么？

  • 边界隔离 — 数据库结构变了，只改 PO 和转换层，不影响上层业务
  • 职责清晰 — Controller 不知道数据库长什么样，Storage 不知道接口返回什么
  • 变更可控 — 重构某一层时，只要接口不变，其他层完全不动

不是为了让开发更复杂
而是为了让长期维护更简单

第三层

中间件深水区

JVM 调优 Redis 数据结构 MySQL 索引 MQ 消息模型 RPC 协议
"技术能力 = 懂得多 + 记得住。我知道的底层原理越多，技术就越强。"
第三层

这个阶段的陷阱

  • 懂原理 ≠ 会用好 — 知道五种数据结构，但场景判断需要的是经验不是背诵
  • 技术深度容易偏离业务 — 花一周研究 JVM 调优，一年可能用不上一次
  • 过度学习边际递减 — 跳表、MVCC 细节对 99% 业务开发是"锦上添花"

真正的技术能力：在合适的场景选择合适的方案
并清楚知道选择的代价和收益

第四层

领域设计初见端倪

从 "这个技术怎么用" 变成 "这个业务需要什么能力"

Conan/
├── conan-commerce-order/   # 交易域
├── conan-commerce-product/ # 商品域
├── conan-fulfill/          # 履约域
├── conan-third-pay/        # 支付域
├── conan-user-center/      # 用户域
└── conan-growth-*/         # 增长域

结构不是技术框架决定的，而是业务边界决定的

第四层

充血模型 vs 贫血模型

public class Order {
  private BigDecimal price, paidFee, refundedFee;

  /** 校验支付价格是否合法 */
  public boolean isInValidPaidFee() { ... }

  /** 订单是否有支付行为 */
  public boolean hasPayAction(Payment p) { ... }

  /** 是否允许退款 */
  public boolean isAllowToRefund(List<Refundment> r) { ... }
}

领域对象不只是数据容器，还包含业务逻辑

第五层

领域本质 — 看见业务全貌

售前 → 售卖 → 下单 → 支付 → 履约 → 交付

商品域

管理"卖什么" — SKU、价格、上下架

交易域

管理"怎么买" — 优惠、订单状态流转

支付域

管理"怎么付" — 支付渠道、状态同步

履约域

管理"怎么交付" — 学期、老师、内容

第五层

领域间的协作边界

  • 交易域不关心商品具体内容，只关心"这个 SKU 能不能卖、卖多少钱"
  • 支付域不关心用户买了什么，只关心"这笔钱收没收到"
  • 履约域不关心订单怎么创建的，只关心"支付成功后我该交付什么"

技术方案是服务于领域边界的，而不是反过来

没有领域意识时容易犯的错误

错误 1：在错误的地方加缓存

✘ "接口慢，加个 Redis 缓存吧"
→ 数据不一致、缓存穿透、维护成本飙升

✔ 先问"为什么慢"，可能需要优化查询或调整领域边界

错误 2：业务逻辑放错位置

✘ 退款判断写在 Controller 里
→ 多入口重复、逻辑不一致、难以测试

✔ 封装在 Order 领域对象：order.isAllowToRefund()

错误 3：跨域直接访问数据

✘ 履约服务直接查订单表
→ 领域边界被打破、数据模型耦合

✔ 通过订单服务接口获取，只依赖接口契约

错误 4：用技术方案代替领域设计

✘ "发短信、Push、积分……用 MQ 解耦！"
→ 消息满天飞、处理顺序不可控

✔ 先想清楚谁负责编排，用领域事件而非零散消息

共同点：只关注技术实现，不关注业务本质

什么是真正的服务端能力

  • 技术可以学，框架会过时，但领域理解是你的护城河
  • Redis 可以换缓存方案，Java 可以换 Go/Rust
    但"履约流程怎么运转""库存需要哪些边界"不会因技术栈变化而失效
  • 领域专家比技术专家更稀缺 — 会写 Redis 代码的人很多，
    理解"为什么这个场景需要这样设计缓存策略"的人很少

技术深度应该服务于领域理解，而不是脱离业务自嗨
最强的服务端工程师，是既懂技术又懂领域的人

五层认知对比

阶段 核心认知 典型问题 突破点
1. CRUD 万能论三层架构搞一切复杂后难维护接触架构规范

3. 中间件深水区技术深度 = 能力懂但不会用思考"为什么用"

CRUD 万能论 → 架构困惑 → 中间件深水区 → 领域设计 → 领域本质

你在哪个阶段？

问自己 答"否"则
我能解释为什么项目要用这种分层结构吗？还在第一层
我能说清各种 O 存在的价值，而不只是"规范要求"吗？还在第二层
用 Redis/MQ 时，我能说出"为什么用"而不只是"怎么用"吗？还在第三层
我能画出业务涉及哪些领域、它们如何协作吗？还在第四层
我能解释公司业务的完整链路、以及自己的位置吗？走向第五层

针对不同阶段的建议

第一层：CRUD 舒适区

主动接触复杂项目
遇到难维护先想分层问题
读懂好项目的目录结构

第二层：规范困扰

理解规范在解决什么问题
维护 2 年以上系统体会分层价值
故意不分层感受后果

第三层：沉迷技术

每学一个技术点问：什么场景会用？
答案是"面试"就往后放
花同样时间理解业务收益更大

第四层 & 第五层

参与跨域需求、画领域地图
多和产品/业务聊"为什么"
沉淀领域知识、帮助新人

核心原则：螺旋上升

技术深度和领域宽度需要交替螺旋上升

技术深度 Lv.1 → 领域理解 Lv.1
↓
技术深度 Lv.2 → 领域理解 Lv.2
↓
技术深度 Lv.3 → 领域理解 Lv.3

每一次技术深入，都应该是为了解决当前领域问题
每一次领域拓展，都会暴露新的技术瓶颈

核心 Takeaway

服务端的本质不是技术
而是领域

技术是你实现领域逻辑的工具。
工具可以换，但对领域的理解是你真正的核心竞争力。
带着这个认知去学技术，你会更清楚每个技术点存在的意义。
