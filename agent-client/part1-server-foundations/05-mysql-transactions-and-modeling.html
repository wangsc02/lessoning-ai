先分清两层建模

领域建模
• 业务上有什么对象、状态、动作
• 更关心业务语义是否清楚
• 产物：业务对象、状态机、规则边界
vs
数据建模
• 怎么落成表、字段、唯一键、索引
• 更关心存储和查询是否可靠高效
• 产物：表结构、主键、状态字段、索引

两者经常相关，但不一一对应：一个业务对象可能对应多张表，一个业务动作可能改多条数据

领域建模

先问清楚这些问题

切入方式

  • 这个业务里真正被管理的"东西"是什么
  • 它是长期存在的，还是一次性过程数据
  • 调用方关心对象本身，还是对象的某种状态

例：学习计划场景

  • 学习计划本身
  • 计划状态 PENDING / CONFIRMED / INTERRUPTED
  • 学习进度（层、周、节）
  • 周计划版本、变更日志

所谓"一个功能"，实际往往不是一个对象，而是一组相关状态

领域建模

状态设计 + 业务动作

状态设计很容易被低估

  • 工程代码看起来：创建、更新、查询
  • 业务语义上：有效/无效、已读/未读、已存在/不存在
  • 这些状态不是数据库顺手带上的，它们本身就是业务的一部分

业务动作比单张表更重要

  • 不要只盯着表：更新 plan、插入 version、记录 log
  • 真正的动作：确认计划、修改长期计划并追加版本、中断计划
  • 这些动作才是应该先被理解、先被保护的
第二部分

再讲数据建模

数据库里到底
怎么承接它

数据建模

数据建模不是存字段，而是承接业务

元素常见作用
主键标识一条记录
唯一键防重、幂等、避免重复创建
状态字段承接业务状态变化
时间字段表达创建、更新时间，支持排序和排查
索引控制查询代价和访问路径

一个业务对象的完整语义，未必都在一张表里（主表 + 版本表 + 日志表）

数据建模

查询方式会反过来塑造数据模型

String sql = "SELECT " + COLUMNS + " FROM " + TABLE +
    " WHERE userId = :userId AND effectiveDate <= :date " +
    " ORDER BY effectiveDate DESC, id DESC LIMIT 1";
  • 为什么要按 effectiveDate DESC, id DESC 双字段排序
  • "当前有效版本"本身就是一个业务查询语义，不只是查最新一条
很多人先建表后写查询，但真实项目经常反过来：要支持怎样的查询，最后会倒逼你怎样建模
第三部分

最后讲事务与一致性边界

这个业务动作
最少要保护到什么程度

事务

事务到底在保护什么

事务最容易被误解成一个抽象定义题。在工程里，它更像是在回答：
这个业务动作，最少要保护到什么程度，才不至于把系统状态写坏

学习计划这条链上的动作：

确认学习计划 修改计划并追加版本 中断计划 状态变化后补推

哪些必须一起成功，哪些允许稍后补上，哪些靠幂等重试兜住 — 这才是事务要回答的问题

事务

几种常见手段，不是在解决同一个问题

手段更适合保护什么
数据库事务同一个本地数据库边界里的原子性
应用层锁并发竞争下的串行化执行
条件更新防止状态被错误覆盖
事件驱动后的最终一致性主数据写完后再推进派生数据
事务

代码示例：confirmPlan

boolean success = studyPlanService.update(userId, updateParam);
if (success) {
    plan.confirm();
    ScheduleVersion version = plan.addScheduleVersion(schedule, effectiveDate, ...);
    studyPlanService.addScheduleVersion(userId, version);
    changeLogService.logLongTermChange(userId, beforeSnapshot, afterSnapshot, ...);
    publishLongTermPlanStatusChanged(userId, plan.getLessonId());
}

事务真正要保护的，不是"几条 SQL"，而是"确认计划"这个动作本身

为什么不是"包大事务"就更安全？边界过大会导致：锁持有时间变长、可恢复性变差、错误原因更难定位

↓

业务动作先于表操作 · 事务边界贴着动作来定 · 一个业务对象不一定只靠主表承接 · 数据库内一致性和后续流程一致性不是一回事

常见误区

五个典型问题

把建模理解成"先把表建出来"

导致后面所有语义都只能往表上硬贴

把事务理解成"能包就都包进去"

通常不是更安全，而是更难维护

把索引理解成"性能差了再补"

很多查询代价在接口设计刚出来时就已经注定

把状态字段、唯一键当成实现细节

这些往往正是业务约束落地的地方

把领域建模和数据建模混成一件事 → 业务对象被表结构牵着走，表结构又承接不了真实业务动作

希望形成的感觉

  • 写服务端需求时，不只看接口和代码，还会先想业务对象和状态
  • 设计表结构时，不只想字段怎么存，还会想唯一约束、状态字段、索引和查询代价
  • 设计事务边界时，不只想"包没包起来"，还会想它到底在保护哪个业务动作
