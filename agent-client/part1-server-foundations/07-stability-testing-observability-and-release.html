大多数线上事故，不是代码逻辑写错了，而是没人知道它已经错了
测试

核心问题：上线后我敢不敢不盯着？

第一层：单测

核心逻辑，下次改了能立刻知道

第二层：集成

多模块连起来是否正确

第三层：边界

空值、重复、并发、超时

大多数线上问题出在"异常流程没人想过"

稳定性

逻辑没错，为什么上线还是出问题？

答案：运行时条件超出预期

  • 下游接口平时 50ms，今天 5 秒
  • 重试逻辑正常情况没问题，下游挂了后把对方打死
  • 流量突然翻 10 倍
  • 慢查询占住连接池，其他请求全超时

这些都不是 bug，但影响可能比 bug 严重得多

稳定性

四个基本防线

超时

任何外部调用都必须设超时
不设 = 线程被无限期占住

重试

要幂等、要递增间隔、要有上限
否则可能把下游打挂

降级

区分核心和非核心
非核心挂了不该影响整体

资源保护

线程池隔离、连接池保护
一个慢请求不能拖垮全部

不是"高级功能"，而是服务端代码的基本功

可观测性

出了问题，先看什么？

很多人第一反应是翻日志 → 在几十万条记录里大海捞针

指标 Metrics
→
Trace 链路
→
日志 Logs
工具回答什么什么时候看
指标系统整体是否正常第一时间判断方向
Trace哪个节点出了问题缩小范围，定位节点
日志具体发生了什么最后确认根因
上线

不只是合并代码

代码合并 → 上线前检查 → 灰度发布 → 上线后观测

上线前检查

配置就绪？依赖就绪？数据兼容？发布顺序？

灰度发布

1% → 10% → 50% → 100%
问题影响范围可控

上线后观测

错误率、P99、业务指标、资源消耗

回滚

不是丢人的事，是最基本的安全网

稳定性是后面再加的事

上线 = 部署成功

灰度+观测+确认才算完成
