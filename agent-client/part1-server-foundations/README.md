# Part 1：服务端基础与 Java 工程体系

> **TL;DR**：这一部分是整套学习文档的核心，目标不是"知道很多后端名词"，而是 **1 个月内具备参与 Java 服务端开发的最低可用能力**。内容压缩成 `7` 篇必修文，外加 `2` 篇可选扩展，优先保证"学完能用"，而不是"覆盖得全"。

## 为什么是这 7 篇

如果要让偏客户端/前端的同学在 1 个月内真正具备服务端开发能力，最重要的不是追求知识点最多，而是先打通一个最小闭环：

1. 知道服务端到底在解决什么问题
2. 知道一次请求在系统里怎么流动
3. 知道接口应该如何设计和约束
4. 知道 Java 和 Spring 如何承载这些约束
5. 知道数据和中间件为什么存在
6. 知道功能做完后怎么验证、观测、上线

这 7 篇就是围绕这个闭环组织的。

## 必修 7 篇导读

### 1. 服务端认知的五层觉醒：从 CRUD 到领域

**目标读者**：刚开始从客户端/前端转服务端的研发同学。  
**TL;DR**：服务端不是技术栈的堆砌，而是对业务领域的深度建模。大多数工程师会经历"CRUD 万能论 → 架构模式困惑 → 中间件深水区 → 领域设计 → 领域本质"五个认知阶段。最终你会发现：领域才是服务端工程师需要掌握的最核心技术。

**核心问题**
- 服务端到底在解决什么问题？
- 为什么"接口通了"不等于事情做完了？
- 为什么服务端更强调边界、失败、资源占用和长期维护？

**验收标准**
- 读者能解释服务端的核心约束：状态、一致性、失败、成本
- 读者能说清为什么服务端问题经常发生在"功能之外"

### 2. 服务端链路地图：一次请求到底经历了什么

**目标读者**：已经知道接口是什么，但对服务端整体链路没有地图的同学。  
**TL;DR**：服务端知识之所以容易学散，通常不是因为知识点太多，而是因为脑子里没有一张"请求链路地图"。把一次请求从入口到返回走通，Java、Spring、DB、缓存、日志、异常处理就都能串起来。

**核心问题**
- 浏览器、客户端或前端发起请求后，服务端到底发生了什么？
- 网关、Controller、Service、Repository 分别在做什么？
- 日志、缓存、数据库、异常、返回值是在链路哪个位置发挥作用？

**验收标准**
- 读者能手画出一条典型请求链路
- 读者知道常见代码分层大致落在哪个环节

### 3. 接口契约：HTTP、RPC 与服务端协作边界

**目标读者**：已经开始接触接口开发，但对"接口设计好坏"没有清晰判断标准的同学。  
**TL;DR**：第三篇真正要讲的不是 HTTP 语法，也不是某个 RPC 框架，而是接口之间的约束与边界。HTTP 和 RPC 只是两种不同的承载形式，真正决定协作成本的，是成功与失败如何表达、超时与重试如何处理、幂等如何保护状态、接口如何兼容演进。

**核心问题**
- HTTP 和 RPC 为什么都属于接口契约的一部分？
- 成功与失败、超时与重试、幂等与兼容分别在保护什么？
- 为什么"能联调"不等于"接口设计合理"？

**验收标准**
- 读者能判断一个接口设计是不是"只可联调，不可维护"
- 读者能解释错误分层、重试、幂等、兼容这些概念的实际工程价值

### 4. 服务端代码表达与工程组织：Java、Spring 与分层边界

**目标读者**：已经会写一点 Java，也接触过 Spring，但离"能读项目代码、写业务代码、判断分层好坏"还差一段距离的同学。  
**TL;DR**：这一篇把 Java 和 Spring 放在同一个工程语境里讲。Java 负责把类型、输入输出、异常路径和业务意图说清楚，Spring 负责把入口层、流程层、存储层和依赖方向组织清楚。重点不是语法点和注解名，而是代码有没有把边界和职责表达清楚。

**核心问题**
- 为什么会写 Java，不等于会写服务端代码？
- 为什么会用 Spring，不等于知道怎么组织系统？
- 类型、方法签名、异常、返回值、分层和依赖方向分别在表达什么？

**验收标准**
- 读者能把"对外接口边界"和"进程内分层边界"明确区分开
- 读者能判断一段逻辑该放在哪一层，以及某个抽象是否真的有价值

### 5. MySQL、事务与数据建模基础

**目标读者**：已经能看懂服务端代码，但对数据层理解还停留在 CRUD 的同学。  
**TL;DR**：这一篇不把数据库当成一个孤立技术点，而是把它放回服务端主线里理解。先分清领域建模和数据建模，再去理解事务、一致性边界、表结构和索引，这样更容易看清为什么很多服务端问题最后都会落到数据层。

**核心问题**
- 为什么"建模"不能只理解成"设计表"？
- 领域建模和数据建模分别在回答什么问题？
- 事务到底在保护什么？它和业务动作、状态变化、唯一约束是什么关系？

**验收标准**
- 读者能区分领域建模和数据建模，不再把"建表"当成建模的全部
- 读者能理解业务动作、事务边界、唯一键、状态字段和查询方式之间的关系

### 6. Redis、MQ 与异步系统的最小必备认知

**目标读者**：开始接触缓存、消息队列，但容易把它们当成"高级组件"的同学。  
**TL;DR**：这一篇不把 Redis、consumer、job 当成三块平行知识，而是把它们放回中台商品服务的同一套系统里理解。数据库仍然承接主状态，但 Redis 已经在承接读路径数据和运行时结构。正因为如此，系统一边需要 consumer 持续同步，一边也必须靠 job 去主动校验、回填和修正。

**核心问题**
- 为什么数据库承接了主状态之后，系统问题还没有结束？
- Redis、consumer、job 分别在补哪一类系统能力？
- 系统的复杂度是怎么一层一层长出来的？

**验收标准**
- 读者能区分 Redis、consumer、job 分别解决什么问题
- 读者能理解多层数据一致性的代价，而不只是把 Redis 理解成缓存
- 读者能判断一个问题更适合实时追赶还是周期修复

### 7. 交付闭环：测试、可观测性与安全上线

**目标读者**：已经理解基本开发流程，但还没有形成"工程交付闭环"的同学。  
**TL;DR**：会写功能，只是服务端能力的一半。另一半是知道怎么验证它是对的、怎么在出问题时快速发现、怎么在运行时保护它不被拖垮、以及怎么安全地把它交付到生产环境。这一篇就是要把"会开发"变成"能交付"。

**核心问题**
- 功能做完后，至少该验证什么？
- 逻辑没错，为什么上线还是会出问题？
- 出了问题，该先看指标、Trace 还是日志？
- 从代码合并到用户用上，中间还有什么事？

**验收标准**
- 读者知道功能做完后至少该如何验证和观测
- 读者理解服务端上线不只是"合并代码"这一个动作
- 读者知道超时、重试、降级、资源保护是基本防线，不是高级功能

## 2 篇可选扩展

### 认证、鉴权与服务端安全边界

**适合什么时候加入**
- 团队的业务对权限、身份、审计、安全要求比较高
- 读者已经开始接触登录态、权限模型、接口安全边界

**为什么不放进首轮 7 篇**
- 重要，但不是所有团队在第一个月里都必须优先消化的主题
- 更适合作为首轮完成后的增强篇

### 为什么 Agent 工程仍然要先补服务端基础

**适合什么时候加入**
- 团队准备从"会做服务端"进一步进入"会做 Agent 工程"
- 需要解释为什么 Agent 反而更依赖契约、观测、测试、运行时

**为什么不放进首轮 7 篇**
- 这篇更适合作为收束篇，帮助团队把服务端基础和后续 Agent 化转型连接起来
- 但如果放得太早，容易让首轮学习目标失焦

## 什么内容被刻意延后

为了严格控制 1 个月的学习负担，下面这些内容暂时不放进首轮必修：

- JVM 深水区与 GC 调优
- 复杂分布式事务
- 微服务治理大全
- WebFlux / 响应式编程
- DDD 全家桶
- K8s 深度运维
- 生产级 Agent 工程体系

这不是说它们不重要，而是当前阶段的目标不是"成为服务端专家"，而是"先能稳定参与服务端开发"。
