服务端链路地图

一次请求的一生：从入口到返回的完整旅程

为什么学服务端会越学越乱？

  • 今天学一点 Spring 注解
  • 明天看一点 MySQL 索引
  • 后天补一点 Redis
  • 再过两天背一背 HTTP 状态码

每个知识点单独看都能理解
但回到项目里，还是有很强的割裂感

根因：缺少一张链路地图

服务端的很多知识点，本质上都在回答同一个问题：

一个请求从外部进入系统之后，到底经过了谁；
每一层分别解决什么问题；出了问题应该在哪一层看。

如果这张图不清楚，学到的框架、组件、规范都会像散装零件
如果这张图清楚了，后面的很多东西都会自动归位

全景：一条请求到底经过谁

v

v

请求链路图不是目录结构图，而是职责流转图

第一个反直觉点

请求一定会经历
"入口治理 -> 协议适配 -> 业务编排 -> 数据访问 -> 响应返回"
这几个基本阶段

命名差异不会改变这个事实
真正重要的不是目录名字，而是职责边界

第 0 站

请求带来了什么

URL

你要访问哪个资源

Method

读取、创建、更新，还是删除

Headers

身份、版本、设备、链路追踪等上下文

Body / Query

真正的业务参数

服务端处理的从来不是一坨参数，而是一个带着上下文的请求

第 0 站

服务端先要判断什么

  • 这个请求是谁发来的
  • 它想操作什么资源
  • 它有没有资格操作
  • 这些参数是不是合法
  • 这个动作在当前状态下是否允许

不是"拿到参数就写逻辑"，而是先判断请求的上下文

第 1 站

入口层：网关的职责

  • 路由 — 把请求转发到正确服务
  • 鉴权 — 先判断请求是否具备访问资格
  • 限流 — 在流量过大时保护后端服务
  • 统一日志 — 记录入口请求的基础信息
  • 灰度与流量策略 — 在不同版本/实例间做流量分配
第 1 站

两类网关的区别

类型 更关心什么 常见职责
接入层网关
(Nginx / SLB)		 流量接入与转发效率 反向代理、负载均衡、TLS 终止、静态资源
业务网关
(API Gateway)		 API 入口治理 鉴权、限流、灰度、统一日志、统一错误包装

网关负责入口治理，但不应该承载具体业务决策

第 2 站

Controller：协议适配层

应该做

  • 接收路径参数、查询参数、请求体
  • 做基础参数校验
  • 调用应用层 / 业务层
  • 把结果转换成统一响应结构

不应该做

  • 承载大量业务判断
  • 编排多个下游依赖
  • 控制事务边界
  • 直接拼复杂 SQL 或操作数据库
第 2 站

Controller 示例：克制的价值

@RestController
@RequestMapping("/api")
public class VersionInfoController {

    @Autowired
    private VersionInfoLogic versionInfoLogic;

    @GetMapping("/version-info")
    public RestResp<VersionInfoVO> get() {
        return versionInfoLogic.get();
    }
}

离请求近，但离业务细节远

第 3 站

Service：业务中枢

  • 参数与上下文进一步校验
  • 业务规则判断与流程编排
  • 协调多个依赖并收束结果

这一层为什么最复杂？
因为系统里的复杂度，本来就主要聚集在这里

第 3 站

Service 示例：业务判断

@Service
public class UserReadStatusLogicImpl {

    public RestResp<Void> clickRedPoint(long userId, int businessType) {
        // 校验请求参数
        if (userId <= 0) return RestResp.error("userId is invalid");
        // 校验上下文一致性
        if (userId != getUserId()) return RestResp.error("...");

        // 组装业务条件对象
        ReadStatusCondition condition = new ReadStatusCondition();
        condition.setUserId(userId);
        // 调用后端服务
        readStatusService.markReadStatusAsRead(condition);
        return RestResp.success();
    }
}
第 3 站

Service 常见误区

Service 是业务中枢，不是万能垃圾桶
  • 什么时候允许这个动作发生
  • 需要改哪些状态
  • 调用顺序是什么
  • 哪一步失败应该回滚，哪一步失败应该补偿

正确的分层不自动保证系统清晰
它只是给了你一个管理复杂度的机会

第 4 站

Repository / Storage：数据访问层

关心什么

  • 根据条件查询数据
  • 插入、更新、删除数据
  • 隔离数据库访问细节
  • 读写分离策略

不关心什么

  • 接口返回结构
  • 页面展示逻辑
  • 复杂业务决策
  • 为什么要执行这个操作
第 4 站

为什么要隔离数据访问

"直接在 Service 里写 SQL，不也能跑吗？"

当然能跑。问题从来不是"能不能跑"，
而是"变化来了之后，系统还能不能稳"
  • 表结构调整时，影响更可控
  • 读写策略调整时，影响更可控
  • 缓存、分库分表、索引优化介入时，影响更可控

Repository 不是为了形式上的分层，而是为了隔离变化

横切关注点

链路的"神经系统"

主链路
Gateway
↓
Controller
↓
Service
↓
Repository
验证与控制
参数校验 · 拦截非法
异常处理 · 统一表达

观测与追踪
日志 · 记录行为
指标 · 量化状态
Trace · 串联链路

横切关注点横跨多层，不属于某一层独占

横切关注点

参数校验：分层处理

协议层校验 (Controller)

字段缺不缺、类型对不对、格式合不合法

越早挡住越好

业务层校验 (Service)

这个动作在当前状态下是否允许

需要结合业务上下文判断

好的校验，不是"都放一层"，而是按职责分层

横切关注点

异常处理：统一表达

  • 哪些异常应该继续往上抛
  • 哪些异常应该转成业务错误
  • 最终用什么方式返回给调用方
异常处理不是为了"把报错藏起来"
而是为了让错误表达稳定、可理解、可排查
横切关注点

事务：放在业务编排层

  • 事务保护的是一组必须一起成功或一起失败的状态变更
  • 更适合放在业务流程被编排的地方 (Service 层边界)

事务打在太底层

只能保护局部
上层流程仍可能一半成功一半失败

事务边界拉得过大

锁持有时间更长
并发能力更差，回滚范围过重

横切关注点

日志：不是出了错才打

真正有价值的日志，至少要回答：

  • 请求来了没有
  • 关键业务动作做了什么
  • 结果是什么
  • 如果失败，失败在什么环节

日志不是"后补的排障工具"
而是系统对自身行为的一种结构化表达

横切关注点

Trace / Metrics：请求返回了不代表结束

在客户端，功能到"页面对了"就告一段落
在服务端，功能到"链路可观测了"才算真正完成

服务端要长期运行，要面对：

偶发超时 下游波动 流量高峰 线上脏数据 环境差异

第二个反直觉点

一条请求真正结束
不是它返回响应的时候
而是它被系统可靠记录、可被解释、可被定位的时候

最小案例：用户点击红点

"用户点击红点，服务端把该状态标记为已读"

请求进入 -> 入口治理 -> Controller -> Service -> Storage -> DB

看起来只是一个"点红点"的小动作
背后仍然经过了完整的链路职责分工

最小案例：每一步做什么

环节 职责
入口层路由、鉴权、限流、记录入口日志
Controller从路径/参数里取值，调用业务层
Service校验参数、校验上下文一致性、组装条件对象、调后端服务
Storage把条件转成 SQL，执行更新，返回结果
响应返回组装统一响应、记录日志、串联 Trace

常见误区

误区 1

接口能返回 = 功能做完了

不代表参数校验合理、错误表达稳定、日志足够定位、链路可观测

误区 2

分层只是套路

不同层在隔离不同类型的变化：协议变化、业务编排变化、数据访问变化

误区 3

日志、监控、Trace 是后补的

这些能力不是上线后补丁，而是链路设计的一部分

误区 4

网关、鉴权、限流不属于主链路

请求不是从 Controller 开始的，很多请求根本到不了 Controller

核心 Takeaway

大多数服务端问题
不是出在某一行代码写错了
而是出在链路中某个职责放错了位置

真正理解链路，学到的是判断力

  • 这段逻辑应该放在哪一层？
  • 这类错误应该在哪一层处理？
  • 这个能力属于业务逻辑，还是入口治理，还是数据访问？
  • 当线上出问题时，我应该先去哪一层看？

有了这张链路地图，后面学 HTTP、Java、Spring、MySQL、Redis、MQ
都不会觉得是在讲不同世界的事情

它们都在服务于同一件事

让一条请求
能够正确地进入系统
正确地流经系统
正确地离开系统
