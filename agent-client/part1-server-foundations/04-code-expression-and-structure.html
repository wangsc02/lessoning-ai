类型最怕什么

同一个对象从入口一路传到数据库，再一路传回来
  • 对外字段改了，内部逻辑跟着抖
  • 数据库存储结构泄漏到业务层
  • 业务语义和展示语义混在一起

入口模型不等于内部业务模型

而是为了减少"这个字段到底代表什么"的猜测

类型系统不是附属品

方法签名在提前声明调用约定

public interface LessonUserMissionService {

    void createOrUpdateStatusToValidIfExist(LessonUserMissionBO userMission)
        throws DuplicateKeyException;

    Optional<LessonUserMissionBO> getValidUserMissionByUserIdAndMissionIdFromReader(
        long userId, long missionId);

    List<LessonUserMissionBO> getValidUserMissionByUserIdAndMissionIdsFromReader(
        long userId, List<Long> missionIds);
}
表达

三种签名，三种语义

签名模式表达的语义
void + exception重点是"做这个动作"，失败时走异常路径
Optional<T>查询单个对象，结果可能不存在
List<T>查询多个对象，没有结果时返回空集合

让调用方在读签名时就有稳定预期

表达

Optional、空集合、异常分别保护什么

表达方式主要保护什么适用场景
Optional<T>"没有这个对象"是正常情况单对象可能不存在
空集合"没有这些数据"是正常情况批量查询没数据
异常继续往下执行已经不合理参数非法、状态非法、系统故障

最容易踩的坑：什么都返回 null，什么都靠异常表达

表达

推荐的表达方式

  • null — 尽量限制在遗留代码或框架边界
  • Optional<T> — 只用于"单对象可能不存在"
  • 空集合 — 用于"批量查询没数据"
  • 异常 — 用于"这条路径不该被当作普通成功分支继续"
表达

for 循环和 Stream：判断标准是什么

判断标准不该是"够不够现代"
而应该是：这段代码到底是在表达规则，还是在炫操作
场景更推荐
简单映射、过滤、收集Stream
中间有多步条件分支、日志、副作用普通 for 循环
需要短路退出、错误处理、分阶段命名普通 for 循环
一眼就能看懂的数据变换Stream

第二部分

再把"组织"理清楚

如何用 Spring
把系统组织成可维护的结构

组织

入口层的职责：接协议，不是接全部业务

应该做

  • 接收请求
  • 提取上下文
  • 做轻量校验
  • 调用下层
  • 把结果翻译成外部协议可理解的形式

不应该做

  • 复杂业务规则
  • 多步状态流转
  • SQL 拼接
  • 跨多个存储对象的编排
组织

入口层承担太多业务的后果

  • 同一个逻辑在 HTTP 和 RPC 入口各写一遍
  • 入口层和业务层的异常语义开始互相污染

名字可以不同，但职责要稳定

谁负责接协议 谁负责编排流程 谁负责业务规则 谁负责持久化

流程层的边界

流程层负责编排，不代表所有业务规则都应该堆在这里

如果某段规则：

  • 会被多个入口复用
  • 本身有明确业务语义
  • 不应该依赖 HTTP / RPC 上下文

那它通常更适合下沉到领域服务或核心服务层

组织

业务层 vs 存储层

业务层
• 先做入参保护
• 用分布式锁保护并发
• 把"创建或恢复状态"作为完整业务动作
• 业务动作完成后发布事件
定义"发生什么才算业务上成立"
vs
存储层
• 接收持久化对象
• 发 SQL
• 返回结果
 
执行"把数据怎么存进去"
组织

存储层不该知道什么

  • 为什么这次写库是"创建"
  • 为什么撞唯一键后要恢复状态
  • 为什么成功后要发事件

这些都应该被收在业务层

组织

Spring 的真正价值

Spring 真正厉害的地方，不是帮你把对象 new 出来
而是帮你把系统结构固定下来

理想的依赖方向：

Controller / RPC Handler
↓
Logic / AppService
↓
Domain Service
↓
Repository / Storage
组织

依赖方向乱了会怎样

Storage 反依赖 Service Controller 直接操作数据库 ServiceA 和 B 双向互调

你会失去这些能力：

  • 看目录就理解职责
  • 替换某层实现而不牵连全局
  • 在代码评审时快速判断"这段逻辑放错层了"
组织

什么时候该抽接口

适合抽接口

  • 确实存在多种实现
  • 对外暴露的是稳定能力
  • 某层需要以接口作为边界，便于替换、扩展或隔离

不必急着抽接口

  • 目前只有一个实现，短期没有替换需求
  • 这个类本身就是局部细节
  • 抽出来后除了多一层跳转没有额外收益

接口是边界工具，不是排版工具

最小工程案例

同一个动作，在不同层表达什么

以 createLessonUserMission 为例

入口层 怎么接这个调用 — RPC 方法签名、判断任务是否存在、翻译成响应结构
业务层 这个动作业务上算什么 — 创建任务、已存在则恢复有效状态、发布后续事件
存储层 这批数据怎么落库 — 插入哪张表、用哪些字段、撞唯一键抛什么异常

同一个需求，不是只写一段代码，而是在不同层分别表达不同类型的信息

常见误区

会写注解 ≠ 分层已经完成

@RestController @Service @Repository

分层真正要看的是：

  • 职责有没有分开
  • 依赖方向有没有守住
  • 异常语义有没有在正确层被解释
常见误区

其他典型问题

入口层越来越胖

Controller 开始承担参数组装、业务分支判断、多次下游调用、数据库存取

什么都返回 null，什么都用异常兜

最隐蔽的表达退化
长期会把调用方训练成：先猜、再试、最后靠日志定位

Stream 用得很现代，但代码读不清楚

服务端代码不是比赛谁更函数式
如果读者看不出"这段逻辑在做什么"，就该回到更直接的表达

抽象很多，但没有一个真正在隔离变化

接口、包装类、转换器越来越多
可能只是在用多一层类来缓冲不确定感

希望形成的感觉

  • 入口层负责接协议，别把业务全塞进去
  • 流程层负责组织调用顺序，别和协议细节缠在一起
  • 业务层负责把规则收住，别只是机械调用几个方法
  • 存储层负责和数据打交道，别偷带业务判断

看方法签名时，更在意这些问题

  • 这里表达的是正常分支，还是异常分支
  • 这里适合用 Optional、空集合，还是明确失败
  • 这个对象到底是在表达接口、业务，还是持久化结构

到这一步，Java 和 Spring 才算真正连在一起了
