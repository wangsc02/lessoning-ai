# 2025：干了些什么，想明白了什么

## 写在前面

又到年底了，照例写点东西。

回看这一年，工作上有些事推进了，有些事卡住了，也有些想法被现实教育了。年初接了供应链团队，推动全栈、落地 DDD、做团队工具；年底家里添了新成员，开始体会什么叫"工作和生活的平衡"。管理上最大的转变是意识到：光自己冲没用，得把团队带起来才行。

以下是这一年的一些思考，不一定对，权当记录。

---

## 一、团队融合：系统架构取决于业务架构

年初工作调整，开始同时负责供应链和电商两个团队。

**顺手的部分**：供应链和电商业务本身结合度高，技术栈相近，协作起来比较自然。

**卡住的部分**：产品团队仍然分离，导致人员调度上有分歧。年初预期的"研发团队合并"计划最终没能落地。

这件事让我真正理解了一句话：**系统架构取决于业务架构**（康威定律的另一面）。

技术团队想合并，但产品侧不合并，需求来源仍然是两条线，那技术合并就是空中楼阁。组织形式、汇报关系、KPI 考核——这些"软"的东西，往往比技术方案更决定最终的系统形态。

**这一年的体会**：想推动架构演进，先看业务组织能不能配合。如果不能，与其硬推，不如先在能影响的范围内做局部优化。

---

## 二、全栈推进：AI 让融合成为可能

虽然组织合并没落地，但在技术融合上有了实质进展：**推动两个团队向全栈方向演进**。

目前的进展：
- 所有 Server 同学都可以承担 toB 前端开发
- 前端同学也可以接手 Server 项目
- 尤其是对技术有追求的同学，在 AI 辅助下表现出很大的进步

这让我看到：**融合的路绝对行得通**。

说起来，我自己 23 年就开始写前端了。那时候没有 AI 辅助，作为一个后端被 React 的思想和开发模式搞得很难受：

- **Hook 的心智模型**：`useEffect` 的依赖数组到底该放什么？为什么放了反而无限循环？
- **闭包陷阱**：明明 state 更新了，`useEffect` 里拿到的还是旧值，debug 半天才反应过来是闭包问题
- **重渲染机制**：不知道什么时候该用 `useMemo`、`useCallback`，要么不用导致性能问题，要么滥用导致代码难看
- **状态管理**：props drilling、Context、Redux，什么场景用什么方案，查了一堆文章还是一头雾水

这些东西对前端同学来说是常识，但对后端来说就是一个个坑。没人带的话，每个坑都要踩一遍才能理解。

现在完全不一样了。AI 可以实时解释"这段代码为什么要这样写"，遇到问题直接问，不用自己翻文档、查 StackOverflow。后端同学写 React，不需要先学几个月前端知识，可以边做边学，AI 帮他补全那些"显而易见但不知道"的东西。

**关键洞察**：全栈的阻力从来不是"技术难"，而是"学习成本高"。AI 大幅压缩了这个成本，让跨栈变得可行。

## 三、技术实践：从理论到落地，从个人到团队

### 上半年：DDD 落地尝试

上半年做了一个 DDD（领域驱动设计）的项目落地尝试。

以前读过 DDD 的书，也在技术分享中讲过，但始终是纸上谈兵。这次真正在项目中实践，才体会到：
- 领域建模的价值在于**让业务语言和代码语言统一**
- 聚合根的边界划分，本质是**一致性边界的决策**
- 很多理论上的"最佳实践"，落地时要根据团队能力和项目阶段取舍

**收获**：从"知道"到"做过"，信心完全不同。

### 下半年：从"自己爽"到"团队可用"

上半年的 DDD 实践，更多是自己主导、自己落地。下半年意识到一个问题：

> 做技术不能只考虑自己爽，要考虑通用性和影响面。

团队十几个人，如果技术实践只有自己能玩转，那价值是有限的。于是下半年调整方向，做了一些**团队定制化的开发实践**：

**1. conan-knowledge：团队开发知识库**

初衷：**团队积累的经验如何低成本地复用？**

架构思路：
- 把日常高频操作封装成命令（如 `/analyze-alarm` 一键分析告警）
- 把代码规范、架构约定沉淀成 Rules，自动注入 AI 上下文
- 基于 Git Hook 自动同步，开发者无感知

**2. conan-scm-meta：供应链业务专家**

初衷：**让 AI 成为懂业务的专家，而不是只会写代码的工具**。

架构思路：
- 通过 Git Submodules 聚合 8 个子系统的代码，解决"代码在哪"的问题
- 手工整理业务知识库（领域模型、业务流程），解决"流程是什么"的问题
- 设计结构化的 AI 工作流（如订单问题排查），引导 AI 按步骤执行

本质是：把老员工脑子里的东西沉淀成 AI 可理解的格式。

**3. 基于 Spring AI 的 Agent 开发探索**

团队在 Agent 开发上走了两条路线：

- **n8n 低代码方案**：基于 n8n 搭建了一套 Agent 工作流。实践后发现，它更适合**快速实验**和**非研发角色使用**（如产品、运营自己搭建简单的自动化流程）。
- **Code First 方案**：对于研发来说，代码的可控性、可调试性、可测试性更重要，更倾向 Code First。

技术选型上，评估了 LangChain（Python）和 Spring AI（Java）。最终选择 Spring AI，原因很简单：**团队维护的 Agent 复杂度不高，业务技术栈是 Java，没必要引入 Python 生态增加维护成本**。

这条路验证下来，对于简单 Agent 场景，Spring AI 完全够用。目前计划用这套方案改造团队的几个存量 Agent：

- **客服助手**：自动回答常见问题，降低人工客服压力
- **供应链取数助手**：自然语言查询供应链数据，不用再写 SQL
- **地推爆单助手**：辅助地推团队快速生成活动方案和话术

用统一的技术栈管理这些 Agent，后续维护成本会低很多。

**这半年的转变**：从"自己做项目"到"做能让团队用起来的东西"。

## 四、管理认知：0→1 靠自己，1→100 靠团队

这一年在管理上有两个重要的认知升级。

### 认知一：一个人的力量有限

以前总觉得"只要我再努力一点，加班再多一点，事情就能做完"。今年真正体会到：**无论怎么加班，一个人都做不完所有事**。

自己的价值在于 **0→1**：找到合适的方向，验证可行性，打个样。

但 **1→100**，必须依托团队的力量。激发大家的能量，才能把事情做得更大、更好。

### 认知二：不是我带得好，是他们本来就优秀

以前觉得自己带团队有一套，能把很多人带好。今年反思后意识到：

> 那些人能成长，主要是因为他们本身就优秀。公司给了优秀的人到我这里，我只是发挥了他们的特长，没有压制他们而已。

不是每个人都能被"带好"。有些人需要的是空间和信任，不是指导；有些人需要的是明确方向，不是自由发挥。

**自己不是万能的**。接受这一点，反而能更平和地看待团队管理。

### 认知三：靠谱是换取机会的货币

今年观察团队里不同人的成长轨迹，发现一个规律：

成长快的人，往往不是能力最强的，而是**最靠谱的**。

逻辑是这样的：靠谱的人会被信任，被信任就会得到更多机会，机会多了成长自然快。这是一个正向飞轮——靠谱带来信任，信任带来机会，机会带来成长，成长让人更靠谱。

反过来，如果一个人不够靠谱，就很难进入这个循环。没人愿意把重要的事交给他，他只能做边缘性的工作，能力也就停在那里。想靠自己突破，其实挺难的——因为没有机会去证明自己，也没有足够的挑战来逼自己成长。

**所以对团队同学的建议**：先把手头的事做靠谱，机会自然会来。别急着证明自己有多强，先证明自己值得信任。
